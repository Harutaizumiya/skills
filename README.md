# architect

<div align="center">

### 把陌生项目快速梳理成一份清晰的结构文档

分析现有代码库、识别核心模块、推断模块关系，并生成一份简洁可读的 `docs/structure.md`。

</div>

---

## 介绍

`architect` 是一个面向代码库分析的 Codex Skill。

它会优先从真实项目材料出发，比如：

- `README`
- 包管理与构建配置
- 启动入口
- 路由与主流程文件
- 顶层目录结构

然后将项目拆解成职责清晰、边界尽量不重叠的模块，并整理出一份标准化结构文档，帮助你更快理解一个项目“做什么、由哪些模块组成、数据怎么流动”。

---

## 它能做什么

- 分析一个已有项目的整体架构
- 识别核心能力与主要入口
- 归纳完整的模块划分
- 推断模块之间的数据流和依赖方向
- 自动生成 `docs/structure.md`

---

## 产出内容

执行后会在当前项目根目录生成或覆盖：

```text
docs/structure.md
```

文档固定包含以下几部分：

- `Overview`：项目目标与整体架构形态
- `Modules`：模块划分与职责说明
- `Data Flow`：请求、数据或控制如何在模块间流动
- `Next Steps`：高价值的后续确认项

- “帮我梳理这个项目的模块结构”
- “分析一下这个仓库的架构”
- “生成一份项目结构说明文档”
- “总结这个项目的模块关系和数据流”

---

## 工作方式

`architect` 的默认分析流程是：

1. 找出项目入口和主要产品形态
2. 提取项目实现的核心功能
3. 将这些功能归纳成职责独立的模块
4. 校验模块集合是否覆盖主要能力
5. 推断模块之间的依赖和数据流
6. 输出 `docs/structure.md`

它不会优先依赖猜测，而是尽量基于仓库中的真实文件进行判断。

---

## 适用场景

- 接手陌生代码库时快速建立全局认知
- 给团队补一份结构化项目文档
- 在重构前先确认当前模块边界
- 在做技术评审前先整理架构脉络

---

## 使用原则

这个 skill 的分析偏向“少读但读准”：

- 优先查看最关键的文件，而不是全仓库漫扫
- 模块边界按职责划分，而不是只按目录名划分
- 小的实现细节通常会并入上层模块
- 如果仓库信息不完整，会保守推断，避免过度脑补

## 示例

一个典型结果会像这样：

```markdown
# Project Structure

## Overview
这是一个以 Web API 为核心的项目，整体采用分层结构，将接入、业务、数据访问和外部集成分离。

## Modules
- 接入层：处理路由、参数校验和请求分发
- 业务层：实现核心业务规则与用例编排
- 数据访问层：负责数据库读写与持久化
- 集成层：对接第三方服务与异步任务

## Data Flow
请求从接入层进入，业务层编排流程并调用数据访问层；涉及外部依赖时由集成层完成交互，最终结果返回到接入层输出。

## Next Steps
- 校验模块边界是否与真实代码职责一致
- 补充关键入口文件和主要依赖说明
```

---

## 总结

如果你希望把“读源码建立认知”这件事标准化，`architect` 的价值就在于：用尽量少的上下文扫描，生成一份足够清晰、可继续迭代的项目结构文档。
